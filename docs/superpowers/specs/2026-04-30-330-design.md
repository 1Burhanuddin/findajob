# #330 — Configure-later Gmail integration via app password + IMAP

**Issue:** [#330](https://github.com/brockamer/findajob/issues/330)
**Milestone:** v0.9 — Multi-Tenancy Foundations
**Status:** Brainstormed 2026-04-30; ready for implementation plan.
**Decision posture:** clean break from OAuth — no migration code, no bridge release.

---

## 1. Problem & primary outcome

findajob can ingest LinkedIn (and, via a configurable allowlist, other) job-alert
emails from a user's Gmail. The integration must be **self-service** — the
operator does not click a button or run a command on the user's stack to enable
it — and must minimize the open-source maintainer's upstream operational
responsibility (no GCP-verified app, no CASA security assessment, no shared
OAuth client serving testers' mailboxes).

The current OAuth-based wiring (`config/gmail_oauth_client.json` +
`config/gmail_token.json`, `scripts/gmail_auth.py`) does not meet these
constraints: `gmail.readonly` is a Google **restricted** scope, which means a
shared production OAuth app would require a CASA security assessment
($15–75K) — out of reach for an open-source project. The status quo of "tester
reuses operator's OAuth client" is a temporary workaround that does not scale
and is not a defensible long-term posture.

This spec replaces OAuth with **Gmail app password + IMAP**. App passwords
remain supported by Google for personal Gmail accounts in 2026 (verified
against `support.google.com/accounts/answer/185833` on 2026-04-30); they are
distinct from the deprecated *Less Secure Apps* mechanism.

## 2. Out of scope

- **Onboarding-time Gmail setup.** Onboarding stays paste-back-only; Gmail is
  configured post-onboarding, after the sentinel is written.
- **Indeed alert ingest.** Dropped in #274 (0.08% application rate). LinkedIn
  is the default sender; the allowlist permits any sender at runtime.
- **Service-account-based Gmail access.** Personal Gmail accounts cannot
  delegate to service accounts; only Workspace domains support that.
- **Forwarding rules in the user's Gmail** (e.g. routing alerts to a dedicated
  label). Document but don't automate.
- **First-run wizard mode.** Out of scope; configure-later is the contract.
- **Migration tooling for existing OAuth users.** Decided against; see §10.
  The operator manually re-onboards their own stack and alice's via the new
  flow post-merge.
- **Detection of company rejection ("not selected") emails.** Filed as #362 —
  uses the same IMAP integration, separate spec/plan.

## 4. Transparency contract

This is the load-bearing trust commitment to users. Every implementation
decision in this spec must honor it; the test suite codifies enforcement.

1. **Read-only IMAP verbs only.** The codebase uses only `LOGIN`, `LIST`,
   `SELECT`, `SEARCH`, `FETCH (BODY.PEEK[])`, and `LOGOUT`. No `STORE`,
   `COPY`, `EXPUNGE`, `APPEND`, `MOVE`, `CREATE`, `DELETE`, or `SUBSCRIBE`.
2. **`BODY.PEEK[]` only.** Messages remain in their pre-fetch `\Seen` state.
3. **Server-side sender filtering.** The `SEARCH (FROM ...)` filter is what
   limits scope — findajob never fetches messages outside the user's
   `sender_allowlist`.
4. **No SMTP code path.** No `smtplib` import anywhere in `src/findajob/`,
   no outbound mail capability of any kind.
5. **Credentials never leave the stack.** The `app_password` is never passed
   to `log_event`, `audit_log`, `notify.send_*`, the LLM client, RAG indexing,
   error reporters, or any HTTP call other than `imap.gmail.com:993`.
6. **No telemetry of message contents.** Raw bytes never go to any external
   service. Parsed `(title, company, url)` tuples land in the local `jobs`
   table; that table is the existing one and is not exfiltrated either.
7. **Both creds files gitignored.** `config/gmail.json` and
   `config/gmail_state.json` covered by `.gitignore` and the local pre-commit
   hook's path patterns.
8. **Audit links pin to the running build's git SHA.** The disclosure banner
   in `/config/gmail/` links to the exact commit deployed, not `main`.

Enforcement: `tests/test_transparency_invariants.py` codifies items 1, 2, 4,
5, and 7 as executable assertions. Items 3, 6, and 8 are enforced by
implementation review (see §11).

### Disclosure banner — full disclosure (expanded)

The collapsed `<details>` block expands to four sections:

1. **Exact scope (what we touch)**
   - IMAP `LOGIN` to `imap.gmail.com:993`
   - IMAP `SEARCH (FROM "<sender>")` for each allowlisted sender, with
     `UID > <last seen>`
   - IMAP `FETCH (BODY.PEEK[])` of message bodies returned by that search
   - IMAP `LOGOUT`
   - That is the entire wire protocol. No `STORE`, no `COPY`, no `EXPUNGE`,
     no `APPEND`, no folder traversal beyond the default INBOX.

2. **What we *can't* do** (even though IMAP technically allows it)
   - findajob has no SMTP code path. It cannot send mail.
   - findajob never calls `STORE` / `MOVE` / `EXPUNGE`. It cannot modify,
     label, or delete your messages.
   - findajob never reads outside your allowlisted senders. The
     `SEARCH FROM` filter is applied server-side by Gmail before anything
     is fetched.

3. **Where your credentials live**
   - `config/gmail.json` on this stack only (chmod 600). Bind-mounted to
     the host filesystem at `state/config/gmail.json`. Never transmitted
     off this machine. Never logged. Never sent to any LLM, scoring
     service, or external API.
   - The repo's `.gitignore` excludes `config/gmail.json` and
     `config/gmail_state.json`. They cannot be accidentally committed.

4. **How to revoke access**
   - **At Google (instant, total revocation):**
     [`myaccount.google.com/apppasswords`](https://myaccount.google.com/apppasswords)
     → find the app password labeled `findajob-<your-handle>` → Revoke.
   - **In findajob (instant, this stack only):** Click *Disconnect Gmail
     integration* below. Wipes both config files. Google-side app password
     remains valid until separately revoked there.
   - **Recommendation:** revoke at Google for any "I want this to stop
     now" scenario; Disconnect for "I want to reconfigure".

The disclosure language is rendered from a Jinja partial
(`src/findajob/web/templates/_gmail_disclosure.html`) used by both the
form page and `docs/setup/gmail.md` (see §9).

## 6. IMAP client module

**File:** `src/findajob/gmail_imap.py` — pure-function-level API, no
FastAPI imports. Importable by `fetchers.py`, the `/config/gmail/` test
endpoint, and unit tests.

### Public API

```python
"""Gmail IMAP client for findajob.

Read-only, app-password authenticated. The only IMAP verbs called are
LOGIN, LIST, SELECT, SEARCH, FETCH (BODY.PEEK[] — does NOT mark messages
read), and LOGOUT. No STORE, COPY, EXPUNGE, APPEND, MOVE, CREATE, DELETE,
or SUBSCRIBE. See docs/superpowers/specs/2026-04-30-330-design.md §4 for
the full transparency contract.
"""

@dataclass(frozen=True)
class GmailConfig:
    address: str
    app_password: str  # spaces stripped on load
    sender_allowlist: list[str]
    configured_at: str  # ISO-8601

@dataclass(frozen=True)
class GmailState:
    last_uid: int = 0
    last_uidvalidity: int = 0
    auth_failure_streak: int = 0
    last_fetched_at: str | None = None
    last_login_at: str | None = None
    last_error: str | None = None

class TestResult(Enum):
    SUCCESS = "success"
    AUTH_FAILED = "auth_failed"
    CONNECTION_ERROR = "connection_error"
    INVALID_CONFIG = "invalid_config"

@dataclass(frozen=True)
class FetchOutcome:
    result: TestResult
    messages: list[tuple[str, bytes]]  # (sender, raw_RFC822_bytes)
    new_uid: int | None
    new_uidvalidity: int | None

def load_config() -> GmailConfig | None: ...
def save_config(config: GmailConfig) -> None: ...
def load_state() -> GmailState: ...
def save_state(state: GmailState) -> None: ...
def test_login(config: GmailConfig) -> TestResult: ...
def fetch_new_messages(config: GmailConfig, state: GmailState) -> FetchOutcome: ...
```

### File mechanics

- **Path constants:** `GMAIL_CONFIG_PATH = f"{BASE}/config/gmail.json"`,
  `GMAIL_STATE_PATH = f"{BASE}/config/gmail_state.json"`. No env-var
  override; users edit via `/config/gmail/`, never directly.
- **Atomic writes:** `save_config()` and `save_state()` write to
  `<path>.tmp`, fsync, `os.replace()` to final, `chmod 600` immediately
  after rename.
- **Missing-file behavior:**
  - `load_config()` returns `None` if file missing → caller treats as
    Off.
  - `load_state()` returns default `GmailState()` (zeros) if missing →
    cold start.
- **Schema versioning:** Both files include `"_schema": 1`. `load_*`
  validates and returns `None` (config) or default (state) on unknown
  version, with a logged validation error.

### Connection lifecycle

```python
def _connect(config: GmailConfig) -> imaplib.IMAP4_SSL:
    """Returns an authenticated client. Caller MUST .logout() in finally."""
    client = imaplib.IMAP4_SSL("imap.gmail.com", 993, timeout=10)
    client.login(config.address, config.app_password)
    return client
```

- **10s socket timeout** — Test button must respond within ~12s end-to-end.
- **Always `try/finally` LOGOUT** — connections close on exception.
- **No connection pooling** — one connection per call.

### Error classification (`_classify_error`)

| Exception | Result |
|---|---|
| `imaplib.IMAP4.error` containing `AUTHENTICATIONFAILED` or `Invalid credentials` | `AUTH_FAILED` |
| `socket.timeout`, `socket.gaierror`, `ConnectionError`, `ssl.SSLError` | `CONNECTION_ERROR` |
| Other `imaplib.IMAP4.error` | `CONNECTION_ERROR` (treat unknowns as transient) |

### `fetch_new_messages` semantics

1. Connect + login. `AUTH_FAILED` → return immediately; caller handles
   streak.
2. `SELECT INBOX`. Capture `UIDVALIDITY` from response.
3. If `state.last_uidvalidity != response.UIDVALIDITY`: Gmail folder reset.
   Treat as cold start: search by `SINCE 7 days` instead of UID; reset
   `state.last_uid` to highest UID found. Log `gmail_uidvalidity_reset`.
4. For each sender in `config.sender_allowlist`:
   `SEARCH (UID <last_uid+1>:* FROM "<sender>")`. Collect UIDs; deduplicate
   across senders.
5. `FETCH (BODY.PEEK[])` each UID — explicitly PEEK, never `BODY[]`.
6. Track `max_uid` across all senders.
7. `LOGOUT` in `finally`.
8. Return `FetchOutcome` with `new_uid = max(state.last_uid, max_uid_seen)`.

### Caller responsibility (`findajob.fetchers.fetch_gmail_jobs`)

```python
def fetch_gmail_jobs():
    config = gmail_imap.load_config()
    if config is None:
        log_event("gmail_skipped", reason="not configured")
        return []

    state = gmail_imap.load_state()
    outcome = gmail_imap.fetch_new_messages(config, state)

    if outcome.result == TestResult.AUTH_FAILED:
        new_streak = state.auth_failure_streak + 1
        gmail_imap.save_state(replace(state,
            auth_failure_streak=new_streak, last_error="auth_failed"))
        log_event("gmail_auth_failed", streak=new_streak)
        if new_streak == 3:
            notify.send_raw(
                "🔐 Gmail login failed — refresh app password at /config/gmail/"
            )
        return []

    if outcome.result == TestResult.CONNECTION_ERROR:
        # Transient; do NOT increment streak
        log_event("gmail_connection_error")
        return []

    # SUCCESS
    gmail_imap.save_state(replace(state,
        last_uid=outcome.new_uid,
        last_uidvalidity=outcome.new_uidvalidity,
        auth_failure_streak=0,
        last_fetched_at=datetime.utcnow().isoformat(),
        last_login_at=datetime.utcnow().isoformat(),
        last_error=None,
    ))
    log_event("gmail_messages_found", count=len(outcome.messages))

    jobs = []
    for sender, raw_bytes in outcome.messages:
        msg = email.message_from_bytes(raw_bytes)
        for job in parse_jobs_from_email_imap(msg):
            job["source"] = _normalize_sender_to_source(sender, job.get("url", ""))
            jobs.append(job)
    return jobs
```

### Parser adapter

The existing `parse_jobs_from_email()` (`fetchers.py:512`) walks a
Gmail-API-shaped dict (`msg.get("payload")` with nested `parts`). The
IMAP path returns Python `email.message.Message` objects. **New parallel
function `parse_jobs_from_email_imap(message)`** that walks the
`Message.walk()` MIME tree and shares the same downstream HTML-parsing
helper. The Gmail-API-shaped variant is deleted in this PR — there is
no shared maintenance benefit to keeping the API-shaped dict format
alive.

The HTML-parsing core (the BeautifulSoup loop, the `SKIP_LABELS` set,
the URL extraction) factors out into a shared private helper in
`fetchers.py` used by `parse_jobs_from_email_imap`. The `SKIP_LABELS`
set, sender-to-source mapping, and downstream URL/title cleaning rules
are all preserved unchanged.

## 7. State + cadence

- **Cadence:** unchanged. `fetch_gmail_jobs()` is called once per
  `triage.py` run (currently nightly midnight PT, plus the supercronic
  10-min `poll_flags.py` poller). No new cron entry. If realtime
  ingestion becomes a real ask later, file a follow-up to add a separate
  `gmail_poll` entry to `ops/scheduled-jobs.yaml`.
- **State:** UID-based incremental fetch via `config/gmail_state.json`
  (see §6). Stateless / SINCE-based scanning is rejected because a
  multi-day outage would silently lose messages.
- **UIDVALIDITY:** monitored on every `SELECT`; mismatch triggers a
  one-time SINCE-7-day cold-start fallback and resets state.

## 8. Failure modes & user signals

| State | Trigger | Behavior |
|---|---|---|
| Not configured | `config/gmail.json` missing | Single INFO log per triage run (`gmail_skipped`); no warnings, no notifications. Status pill: `Off`. |
| Configured + working | IMAP login + fetch succeeds | Existing `gmail_messages_found` event; advance UID checkpoint. Status pill: `Authorized · last fetched <ts>`. |
| Configured + auth failure | IMAP `AUTHENTICATIONFAILED` | WARNING log per triage run. Increment `auth_failure_streak` in state. **On streak transition `2 → 3`:** ntfy notification "🔐 Gmail login failed — refresh app password at /config/gmail/". Status pill: `Login failed — refresh app password`. |
| Configured + transient failure | Network / timeout / SSL | INFO log (`gmail_connection_error`). Streak NOT incremented (transient errors must not trip the auth-failure ntfy). Status pill stays `Authorized` (don't flap on one bad poll). |

**Recovery:** any `SUCCESS` resets `auth_failure_streak` to 0. No
"back online" notification on recovery — log-only is sufficient.
