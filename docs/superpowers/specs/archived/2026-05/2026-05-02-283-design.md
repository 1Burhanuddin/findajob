# Design — #283: source-strategy briefing + profile-grounded source-config emission in onboarding interview

**Issue:** [#283 — feat(onboarding): derive job-search queries + feed URLs from candidate profile during interview](https://github.com/brockamer/findajob/issues/283)
**Date:** 2026-05-02
**Status:** approved by operator; ready for implementation-plan write-up
**Bundled in same PR:** #215 (injector hardening polish from #148 Task 2 review)
**Fast-follow:** #372 (`target_locations.txt` derivation)
**Already-filed companion:** #407 (interactive Gmail IMAP credential capture during interview)
**To file at session end:** curated RapidAPI feed picker + per-class recommendations + interactive signup walkthrough

## 1. Problem statement

The onboarding interview (`config/roles/onboarding_interviewer.md` + `findajob.onboarding.injector`) collects rich candidate context but the job-source configuration files it emits are weakly grounded in that context. Today's path emits a single `jsearch_queries.txt` whose queries aren't explicitly steered by `## Target Roles` / `## Core Competencies`, and emits no `feed_urls.txt` at all — operators end up hand-editing both post-onboarding. There is also no briefing layer that helps the candidate understand *which* sources to use for their field; this matters because RapidAPI (today's primary recall source) is LinkedIn-skewed and produces near-zero recall for candidates in fields with light LinkedIn presence (skilled trades, social services, regional employers, some healthcare niches).

The result: a fresh pipeline shell whose ingestion sources are weakly aligned to what the candidate actually wants to see. Onboarding ends, the pipeline runs, the candidate hand-tunes everything in week two.

## 2. Goal

Onboarding produces *informed source selection* and *profile-grounded source config* in one pass. The candidate finishes the interview with:

- A clear mental model of the four available sources (paid service, company career-page feeds, Gmail alerts, manual) and which fit their field.
- Explicit choice over which sources are active.
- Source-config files emitted *only* for the sources they chose, populated with profile-grounded content (queries from their target roles + core competencies; ATS feed URLs from their target companies; LinkedIn-alerts setup checklist seeded from their queries).

The bar is field-agnostic: a clinical-pharmacist candidate (light LinkedIn presence, hospital-system target list) and a data-center-operations candidate (heavy LinkedIn presence, FAANG target list) both finish the interview with source configs that work for them.

## 3. Out of scope

Deferred to follow-on issues — not part of this design:

- **Multi-RapidAPI feed picker + signup walkthrough + per-class curation.** Section 0's educational layer mentions "today the pipeline uses one specific RapidAPI service; a future version will help you pick the one that best fits your field." That forward-reference is the only hook in #283; the actual picker, multi-adapter layer, and signup walkthrough are a separate Medium-priority follow-on.
- **Programmatic LinkedIn alert creation.** Gmail OAuth doesn't reach LinkedIn UI; this stays a manual operator action.
- **Interactive Gmail IMAP credential capture during the interview.** Already filed as #407 (depends on #283 + #330). Section C of this issue stays doc-only emission of `linkedin-alerts.md` with a closing step pointing at `/config/gmail/`.
- **`target_locations.txt` derivation.** Same family (profile.md → derived source config), separate complexity profile (filter logic vs source config). Fast-follow as #372.
- **On-demand discoverer trigger at onboarding completion.** Section B's day-1 path uses static `## Target Companies` only; if reduced day-1 quality bites testers, file as a follow-on then.
- **Source-selection manifest file** (`source-selection.json` or similar). YAGNI until `/tools/` or the tuning loop has a concrete consumer.
- **Workday support in `feed_urls.txt`.** Pair with the `findajob.fetchers` Workday adapter when that lands.

## 4. Architecture

### 4.1 Two-layer briefing in the existing 5-phase prompt structure

The source-strategy briefing splits into two layers placed at semantically-correct points in the existing 5-phase flow:

**Educational layer — added to Phase 1 (Orientation).** A new sub-section after the existing orientation paragraphs introduces the four-source taxonomy (paid service / company career-page feeds / Gmail alerts / manual) with cost, coverage, and setup tradeoffs. No profile knowledge required at this point — pure taxonomy. The candidate finishes Phase 1 understanding what the sources are without yet committing to any.

**Decisional layer — new sub-phase 3g (Source selection).** Inserted between current 3f (voice samples) and Phase 4. With knowledge of the candidate's `## Target Roles` and `## Core Competencies` from the gap-fill, the LLM recommends a source mix and asks the candidate to pick using letter-prefixed options (per the prompt's existing convention at lines 26–29 of `onboarding_interviewer.md`).

The candidate's source selection is held in the LLM's own context for use in Phase 5 emission. No machine-readable manifest file — YAGNI until a downstream consumer exists.

### 4.2 Conditional emission

Phase 5 emits the three Section A/B/C file blocks only when the candidate's 3g selection requires them. The decision matrix:

| 3g selection | `jsearch_queries.txt` | `feed-urls.txt` | `linkedin-alerts.md` |
|---|---|---|---|
| `a` only (RapidAPI) | ✓ | — | — |
| `b` only (company feeds) | — | ✓ | — |
| `c` only (Gmail alerts) | ✓ | — | ✓ |
| `a,b` | ✓ | ✓ | — |
| `a,c` | ✓ | — | ✓ |
| `b,c` | ✓ | ✓ | ✓ |
| `a,b,c` | ✓ | ✓ | ✓ |
| `none` (manual only) | — | — | — |

`jsearch_queries.txt` is emitted whenever RapidAPI or Gmail alerts is active because both consume it (RapidAPI calls jobs-api14 with these queries; Gmail-LinkedIn alerts use them as the candidate's saved-search seed text).

Conditional emission is enforced by the prompt; the parser is selection-agnostic. This keeps the parser simple and pushes the discipline into the prompt where it belongs (the Phase 5 emission protocol is the right boundary for "what files exist for this candidate").

### 4.3 Parser change (`src/findajob/onboarding/parser.py`)

Single edit: move `jsearch_queries.txt` from `ALLOWED_FILENAMES` (required) to `OPTIONAL_FILENAMES` (tolerated), and add `feed-urls.txt` and `linkedin-alerts.md` to `OPTIONAL_FILENAMES`.

```diff
 ALLOWED_FILENAMES: tuple[str, ...] = (
     "profile.md",
     "master_resume.md",
     "target_companies.md",
     "business_sector_employers_reference.md",
-    "jsearch_queries.txt",
     "prefilter_rules.yaml",
     "in_domain_patterns.yaml",
     "display_name.txt",
     "timezone.txt",
     "ntfy_topic.txt",
 )

-OPTIONAL_FILENAMES: tuple[str, ...] = ("voice-samples.md",)
+OPTIONAL_FILENAMES: tuple[str, ...] = (
+    "voice-samples.md",
+    "jsearch_queries.txt",
+    "feed-urls.txt",
+    "linkedin-alerts.md",
+)
```

No regex changes (the `<<<FILE: name>>>...<<<END FILE: name>>>` block protocol already handles arbitrary filenames). No new code paths.

### 4.4 Injector changes (`src/findajob/onboarding/injector.py`)

Three areas of change:

1. **Filename-mapping table extension.** Add `feed-urls.txt` → `config/feed_urls.txt` (hyphen→underscore convention, matches `jsearch_queries.txt`) and `linkedin-alerts.md` → `candidate_context/linkedin-alerts.md` (hyphen preserved, matches `voice-samples.md` precedent). Same atomic-write + `.backups/{UTC-stamp}/` backup pattern as the existing 10 files.

2. **Loosened missing-files check.** Today's check hard-errors if any ALLOWED file is absent; post-#283 the (smaller) ALLOWED list no longer includes `jsearch_queries.txt`, so its absence is no longer an error. No conditional logic based on candidate selection — the parser/injector remain selection-agnostic.

3. **#215 hardening polish bundled.** The three small fixes from the #148 Task 2 review land in the same PR.

### 4.5 Emission-consistency warnings

A small post-injection sanity check writes warnings (not errors) to `logs/pipeline.jsonl` under `event=onboarding_emission_anomaly`:

- `linkedin-alerts.md` emitted but `jsearch_queries.txt` absent → broken cross-reference in Section C's checklist.
- `jsearch_queries.txt` emitted but contains zero query lines (after comment-stripping) → signals prompt-LLM drift.

Warnings are non-blocking. They give future debugging a tripwire for prompt-revision regressions without impacting the candidate's onboarding.

### 4.6 Backwards compatibility (`?mode=rerun`)

Existing operators on stacks with the sentinel already touched can re-run via `/onboarding/?mode=rerun`. The injector's existing `.backups/{UTC-stamp}/` flow handles `config/jsearch_queries.txt` and any pre-existing `config/feed_urls.txt` identically — backed up before overwrite. No special-case logic for the new files.

`/onboarding/?mode=rerun` is the migration path for existing operators picking up #283. No re-onboarding is *required* — operators can keep their current source configs untouched and only re-run if they want the new briefing-driven path.

## 5. Section content shapes

### 5.1 Section A — `jsearch_queries.txt` (3-4 word natural phrases, profile-grounded)

The prompt's existing `jsearch_queries.txt` exemplar (lines 692–711 of `onboarding_interviewer.md`) gets revised. Today the prompt's guidance for query generation is implicit; under #283 it becomes explicit profile-grounded:

- **Source signals:** `## Target Role`, `## Core Competencies`, `## Career Summary`. Specifically NOT `## Excluded Categories` (those drive the prefilter, not the queries).
- **Constraint:** 3-4 words per phrase. Single-domain words and compound noun phrases only — no Boolean operators, no quoted strings, no location qualifiers, no seniority modifiers. The prompt explicitly cites the `CLAUDE.md` constraint that 5+ word strings return zero LinkedIn results.
- **Volume:** 8–12 queries. Below 8, recall suffers; above 12, hits the 150-call/month free RapidAPI tier in <2 weeks.
- **Field-agnostic:** prompt includes 3 one-shot examples covering different fields (one tech-flavored, one social-services-flavored, one skilled-trades-flavored) to demonstrate query *shape*, not field taxonomy.

### 5.2 Section B — `feed-urls.txt` (Greenhouse + Lever + Ashby URLs from `## Target Companies`)

- **Source signal:** `## Target Companies / Organizations` block from the candidate's profile (the static, candidate-stated tier-1/2/3 list from sub-phase 3c). On day-1 stacks where `discovered_companies.md` is empty, that's the only signal.
- **Output:** one URL per line. Three accepted shapes (matching the existing fetcher slug regexes in `findajob.fetchers`):
  - **Greenhouse:** `https://boards.greenhouse.io/{slug}` or `https://job-boards.greenhouse.io/{slug}`
  - **Lever:** `https://jobs.lever.co/{slug}`
  - **Ashby:** `https://jobs.ashbyhq.com/{slug}`
- **For each company in `## Target Companies`,** the LLM identifies which ATS the company uses (LLM web-knowledge) and emits the right URL shape. Inline comments are supported — the `_parse_feed_slugs` helper already strips `# {Display Name}` after the URL.
- **Companies the LLM doesn't know an ATS for, or that use unsupported ATSes** (Workday, in-house, SmartRecruiters, Jobvite, etc.): emit a comment line `# {company} — uses {ats} or unknown; not currently supported`. Don't fabricate slugs (this is the same hallucination failure mode #345 just fixed for the discoverer).
- **Volume:** typically 5–15 working URLs per candidate plus a handful of commented-out non-supported companies.
- **Field-agnostic:** same "LLM judgment from candidate's stated list" pattern; no enumerated company taxonomy.
- **Footnote for spec:** if/when Workday (or any new ATS) support lands in `findajob.fetchers`, Section B's prompt needs a paired update to start emitting that ATS's URL shape too.

### 5.3 Section C — `linkedin-alerts.md` (LinkedIn saved-search setup checklist)

A 5–8 step markdown checklist the candidate works through manually in their LinkedIn UI:

```markdown
# LinkedIn job alerts setup

The pipeline reads job-alert emails from your Gmail inbox. To fill that
inbox with useful alerts, set up saved searches on LinkedIn that email
you matches.

## Steps

- [ ] On LinkedIn, go to the Jobs tab and search for one of your target
      roles (e.g., "{first query from jsearch_queries.txt}"). Use the
      "Job alerts" toggle on the search results page to enable email
      alerts for this search.
- [ ] Repeat for each query in `config/jsearch_queries.txt`. LinkedIn
      caps you at ~20 active alerts; pick the highest-recall ones if
      you have more queries than that.
- [ ] Set the alert frequency to "Daily" (more granular than "Weekly",
      less noisy than "Real-time").
- [ ] Confirm the alerts are landing in the Gmail inbox you'll connect
      to the pipeline. Check the spam folder once — LinkedIn job alerts
      occasionally land there on the first delivery.

## Wire up the pipeline's Gmail reader

Once those alerts are firing in your inbox, configure the pipeline's
IMAP reader at `/config/gmail/` so it can ingest them automatically.
That page walks you through generating a Gmail app password and
testing the connection.
```

The closing step is the explicit `/config/gmail/` reference satisfying the acceptance criterion that ties Section C back to #330. Doc-only — no interactive credential capture (that's the #407 follow-on).

If the candidate picked `c` (Gmail) without `a` (RapidAPI), they still get `jsearch_queries.txt` emitted (per §4.2's emission matrix). So the cross-reference to `config/jsearch_queries.txt` from `linkedin-alerts.md` is always valid when `linkedin-alerts.md` exists.

## 6. Prompt content (Phase 1 educational layer)

The Phase 1 educational layer adds this block to `config/roles/onboarding_interviewer.md` after the existing orientation paragraphs:

```
## Job-Source Strategy (read this carefully)

This pipeline can find jobs from up to four different places. Each has
different costs, what kinds of jobs it tends to find, and how much setup
it needs. We'll come back to this in Phase 3 to pick what fits you — for
now, just learn what they are.

1. **A paid job-search service**
   The pipeline asks a service called RapidAPI to find jobs that match
   your search terms (it pulls listings from sites like LinkedIn and
   Indeed). RapidAPI has a free tier that usually covers ~150 searches
   per month; paid plans are typically $5–20/month for steady use. You
   sign up at rapidapi.com and paste a key into the pipeline. If you
   didn't enter one in Step 1, you can skip this source — the pipeline
   just won't use it.
   - Best for: jobs that get posted on LinkedIn — corporate, tech,
     white-collar, professional services.
   - Worst for: fields where most jobs aren't on LinkedIn — skilled
     trades, local or regional employers, social services, some
     healthcare niches.
   - Note: today the pipeline uses one specific RapidAPI service. A
     future version will help you pick the one that best fits your field
     and walk you through the signup.

2. **Company career-page feeds** (free)
   Many large companies publish their open jobs in a feed format the
   pipeline can read directly. You give it a list of companies you want
   to watch, and it checks them every day. No signup, no cost.
   - Best for: anyone with specific target employers in mind.
   - Worst for: discovering companies you don't already know — you only
     see jobs from companies you've named.

3. **Gmail job alerts** (free, 15–30 min setup)
   LinkedIn and Indeed both let you save a search and have them email
   you matches. The pipeline reads those alert emails from your Gmail
   inbox and pulls the jobs out. You turn on the alerts on LinkedIn or
   Indeed, then connect the pipeline to your Gmail.
   - Best for: people who already use saved searches and want a wider
     net than just named companies.
   - Worst for: anyone who'd rather not connect their Gmail.

4. **Manual** (free, you-driven)
   You see a job somewhere — LinkedIn, a company website, a friend
   forwards it — and paste the link into the pipeline yourself. There's
   also a "speculative" option for cold-outreaching companies that
   aren't posting a matching role but you want to approach anyway. No
   setup at all.
   - Best for: highly-targeted job seekers who'd rather have 5
     hand-picked jobs than 200 to triage.
   - Worst for: anyone wanting volume without effort.

You can pick any combination. Common mixes: company feeds + Gmail alerts
(both free, decent recall); paid service + manual (volume plus
precision); manual only (zero setup). We'll discuss what makes sense
for *you* in Phase 3 once we know your target roles.
```

## 7. Prompt content (sub-phase 3g decisional layer)

Inserted between current sub-phase 3f and Phase 4:

```
### 3g. Source selection

Now that we know your target roles, let me suggest a source mix.

Based on what you've told me — {one-line recap of target roles} —
here's what I'd recommend: {one or two of the sources, each with a
short plain-language reason}.

Pick which sources you want active (Manual is always available — no
selection needed for that one):

  a. Paid job-search service (RapidAPI)
  b. Company career-page feeds
  c. Gmail job alerts

Reply with the letters you want (e.g. "b" or "a, b, c"). Reply "none"
if you'd rather start with Manual only.

If you skip RapidAPI (no 'a'), any key you entered in Step 1 just sits
dormant — no cost, no problem. You can always come back and add a
source later by re-running onboarding.
```

The prompt also includes 3 one-shot examples illustrating per-profile recommendation:

- A tech / corporate candidate: recommend `a, b` (RapidAPI for LinkedIn-heavy recall, company feeds for FAANG-class targets).
- A social-work candidate: recommend `b, c` (RapidAPI is weak for the field; company feeds for named non-profits; Gmail alerts for Indeed-skewed sector).
- A skilled-trades candidate: recommend `c` only (RapidAPI is weak; company feeds rarely apply; Indeed alerts via Gmail are the highest-recall path).

These examples illustrate the *shape* of recommendation — pure LLM judgment from `## Target Roles` + `## Core Competencies`, no closed field taxonomy in the prompt.

## 10. Acceptance criteria (consolidated)

Mapped from #283's body plus two new criteria from the reshape. Each maps to at least one §-section of this design.

- [ ] `config/roles/onboarding_interviewer.md` adds explicit Phase steps that read `## Target Roles` / `## Target Companies` / `## Core Competencies` and produce derived source config. *(§4.1, §6, §7)*
- [ ] Onboarding emits up to 3 new file blocks: `<<<FILE: jsearch_queries.txt>>>` (revised), `<<<FILE: feed-urls.txt>>>`, `<<<FILE: linkedin-alerts.md>>>`. *(§4.2, §5)*
- [ ] `findajob.onboarding.parser` + `injector` handle the new file blocks (atomic write, backup, sentinel) — same pattern as the existing 7-8 files. *(§4.3, §4.4)*
- [ ] The interview noticeably improves source-config quality vs. the current path: at minimum, derived queries are 3-4 word natural phrases (LinkedIn-friendly per `CLAUDE.md` constraint). *(§5.1)*
- [ ] `linkedin-alerts.md` checklist's closing step points at `/config/gmail/` and references the #330 IMAP integration, so the LinkedIn-alerts → Gmail → IMAP → pipeline path is fully wired by the end of onboarding even though IMAP credential capture itself is out of scope here. *(§5.3)*
- [ ] Backwards-compat: existing operators (sentinel already touched) get `/onboarding/?mode=rerun` to upgrade their source config without losing manual tweaks. *(§4.6)*
- [ ] Field-agnostic: works for the operator's DC ops profile AND for Alice's social-work profile (no enumerated company lists or industry vocabulary in tracked code). *(§4.1, §5.1, §5.2, §7)*
- [ ] **(NEW)** Section 0 educational layer covers all four sources with cost / coverage / setup tradeoffs in plain language; no API/IMAP/RSS jargon without explanation. *(§6)*
- [ ] **(NEW)** Section A/B/C emission is opt-in based on sub-phase 3g letter selection; manual-only path emits none. *(§4.2)*

## 11. Why this matters now

Beta testing has surfaced the weakness: a fresh stack is only as good as its sources. The operator and Alice both pulled the onboarding flow; both ended up hand-editing source configs because the interview's defaults were too generic. Closing this gap is high-leverage for any future beta tester — and for the eventual open-source path, where the candidate has no operator hand-holding.
