# Multi-tenant operator dashboard for docker.lan stack health — Design Spec

## Issue(s)
- #333 — Multi-tenant operator dashboard for docker.lan stack health

**Date:** 2026-04-30
**Status:** Design approved via brainstorming session; ready for implementation-plan pass
**Related work:** #71 (closed — multi-tenancy discipline), #327 (HTTP Basic Auth perimeter), #344 (multi-tenant scheduler stagger), #338 (parent epic — multi-tenancy + per-tester scaling)

## 1. Context

`docker.lan` runs six findajob deployments side-by-side as of 2026-04-30: operator's own (`brock`) plus five testers (`alice`, `dave`, `judy`, `papa`, `tango`). Each is a separate Dockge stack at `/opt/stacks/findajob-{handle}/` with bind-mounted state at `state/{data,config,logs,companies,...}/`. Every stack runs the same image (`ghcr.io/brockamer/findajob:v0.8` or `:latest`); differences are confined to env vars and mounted state.

There is no aggregated view today. "Is everyone's pipeline triaging cleanly?" is currently answered by per-stack manual inspection — `ssh docker.lan && cat /opt/stacks/findajob-{handle}/state/logs/pipeline.jsonl | jq ...` repeated five times. With the v0.8 multi-tenant rollout still soaking and three Phase-3-NUX testers onboarded the day before this spec was written, the friction is acute.

The deliverable is operator-facing situational awareness — *which stacks triaged today, which are stuck, which threw an aichat error in the last 24h* — without per-stack inspection. This is not paging or replacement for the existing per-stack ntfy alerts; it's a single-glance morning check that answers "anyone broken?" in <2 seconds.

## 2. Objectives

| Role | Metric | Treatment |
|---|---|---|
| **Primary objective** | Operator can answer "is any stack stuck?" without SSH | Single page, all six stacks visible, render <2s |
| **Hard floor — image identity** | Operator's image must remain bit-identical to testers' | Behavior gated by env flag (`FINDAJOB_OPERATOR_MODE=1`) — same image, different invocation |
| **Hard floor — read-only** | Dashboard cannot modify any tester state | No POST handlers, all SQLite reads via `mode=ro` URI |
| **Soft floor — failure isolation** | One broken stack must not crash the whole page | Per-row try/except; broken row shows error; siblings render |
| **Generalization gate** | Diff of tracked files | Zero personal handles, ntfy topics, or operator-specific paths in tracked code |

The image-identity gate is the architectural constraint that keeps this work consistent with how findajob has scaled multi-tenancy so far: one image, per-stack env. Violating it would mean operator's stack diverges from testers', which compounds release coordination cost.

## 3. Scope

### 3.1 In scope

- New Python package `src/findajob/admin/` containing pure-data modules: `stack_discovery.py`, `stack_health.py`, `jsonl_tail.py`. No HTML, no FastAPI imports.
- New route module `src/findajob/web/routes/admin_stacks.py` registered iff `FINDAJOB_OPERATOR_MODE=1`.
- New Jinja2 template `src/findajob/web/templates/admin/stacks_index.html` extending `base.html`.
- One conditional template tweak in shared nav (`_nav.html`) — operator-mode renders top bar with red background as a visual mode indicator.
- One conditional `Admin` link in shared nav, also gated on `operator_mode`.
- Tests under `tests/test_admin_*.py` covering discovery, health gathering, JSONL tail, route, nav rendering. New fixture tree under `tests/fixtures/admin_stacks/`.
- Operator-side compose edit (documented; not committed): add `/opt/stacks:/opt/stacks:ro` mount + `FINDAJOB_OPERATOR_MODE=1` env to operator's `/opt/stacks/findajob-brock/compose.yaml`. Tester compose unchanged.
- `CHANGELOG.md` entry under `[Unreleased]` → `Added`, with a `### Migration required` bullet listing the operator-side compose edits.
- Documentation updates: `docs/setup/install-docker.md` (operator-mode subsection), `CLAUDE.md` (route list + container context table), `CLAUDE.local.md` (operator stack notes).

### 3.2 Out of scope

- Drill-down `/admin/stacks/{handle}/log` page rendering tail-N of a stack's `pipeline.jsonl`. Surfaced via a click-to-copy `ssh ... tail -F` recipe in column 7 instead. Follow-up issue to be filed once operator has used the summary view enough to know what drill-down details actually matter.
- Auto-refresh / HTMX polling. Static-on-load only. Follow-up to be filed if the manual-reload cadence proves wrong in practice.
- Per-stack ntfy alerts on stale stacks. Belongs in `notify.py health-check` extension, not the dashboard.
- Tenant-scoped views (testers see their own status). Explicit decision in issue body — testers are non-technical; deferred indefinitely.
- Cross-stack actions (restart from dashboard, force-triage). Display-only.
- Time-series graphs / historical charts. Snapshot view only.
- Per-stack resource limits or runaway-job protection.
- New auth credential. Reuses existing `FINDAJOB_AUTH_USER` / `FINDAJOB_AUTH_PASS`.

## 4. Architecture

### 4.1 Route registration

The findajob app wires routes via `findajob.web.routes.__init__`, which aggregates each sub-module's `APIRouter` into a single package-level `router` that `app.py` includes once. The conditional registration goes inside that aggregator:

```python
# findajob/web/routes/__init__.py — at the bottom of the file
import os
if os.environ.get("FINDAJOB_OPERATOR_MODE") == "1":
    from findajob.web.routes import admin_stacks
    router.include_router(admin_stacks.router)
```

The module is *not* imported when the flag is unset. Tester stacks have no in-memory route at `/admin/stacks/` — a request returns 404, not 403. Defense in depth.

The aggregator pattern also gives us a natural choice on whether the admin route inherits `require_onboarding_complete` (the dependency wrapped around `/board/`, `/materials/`, `/stats/`). Operator's stack will always have completed onboarding, so the dependency is harmless either way; including it keeps the admin route consistent with the rest of the operator-pull surface. Plan task should add it.

### 4.2 Mount surface

Operator's `/opt/stacks/findajob-brock/compose.yaml` adds two lines:

```yaml
services:
  scheduler:
    environment:
      FINDAJOB_OPERATOR_MODE: "1"
    volumes:
      - /opt/stacks:/opt/stacks:ro
```

Read-only and catch-all. Future tester stacks (`findajob-{new}`) appear automatically without compose edits.

### 4.3 Auth

Inherits the existing `BasicAuthMiddleware` from `findajob.web.auth`. No new credential. The middleware allowlist (`/static/`, `/healthz`, `/favicon.ico`) is unchanged. `/admin/stacks/` and `/admin/stacks/?...` require the same `FINDAJOB_AUTH_USER`:`FINDAJOB_AUTH_PASS` Basic Auth as `/board/`.

This is correct because "operator" is defined as "person with access to operator's stack" — there's no finer-grained identity in the system today and inventing one is out of scope.

### 4.4 Render strategy

Static-on-load. One server-side Jinja2 render per request. No background polling, no cache, no HTMX swap. Browser reload to refresh. Footer line states `Rendered at {ts_utc}. Browser reload to refresh.` to set expectation.

### 4.5 Data-flow per request

1. Auth passes (existing middleware).
2. `discover_stacks(Path("/opt/stacks"))` returns `list[StackPath]` — one entry per `findajob-*/state/` directory, with handle + paths to `data/pipeline.db` and `logs/pipeline.jsonl`.
3. For each `StackPath`, `gather(stack)` returns a `StackHealth` dataclass:
   - SQLite read-only connection (`sqlite3.connect("file:...?mode=ro", uri=True)`); one query for stage distribution and `prep_in_progress > 60min` count.
   - Bounded JSONL tail (last ~1MB) classifying these canonical event names (verified against operator's live `pipeline.jsonl` 2026-04-30):
     - **Triage lifecycle:** `pipeline_started`, `pipeline_complete`, `pipeline_terminated`.
     - **Failures:** `aichat_failure`, `discovery_failed`, `prep_failed`, `prep_failed_reset`.
     - **Discovery:** `discovery_complete`.
   - `last_triage_complete_ts` is sourced from the most-recent `pipeline_complete` event in the JSONL tail (the issue body's AC labels this "triage_complete" — that's the conceptual name; the canonical event name in code is `pipeline_complete`). It is **not** sourced from `audit_log` — `audit_log` records stage transitions on individual jobs, not pipeline-wide cron-event lifecycles.
   - 24h triage success count = number of `pipeline_complete` events in last 24h.
   - 24h triage failure count = number of `pipeline_terminated` events in last 24h.
   - Last-failure timestamp = most-recent of `aichat_failure` / `discovery_failed` / `prep_failed` / `prep_failed_reset`.
   - Freshness bucket computed from `last_triage_complete_ts`: `<24h` green, `24-36h` yellow, `>36h` red (per AC).
4. Template renders one row per stack. Operator's stack first, others alphabetical.

### 4.6 Performance budget

| Step | Per stack | 6 stacks (sequential) |
|---|---|---|
| Glob `/opt/stacks/findajob-*/state/` | <10ms | <10ms (one glob) |
| SQLite open + aggregate | ~50ms | ~300ms |
| JSONL tail + decode | ~30ms | ~180ms |
| Render | — | ~50ms |
| **Total** | — | **~540ms** |

Comfortable inside the 2-second AC budget. Sequential execution; no async/threading needed at six stacks. Revisit at 20+.

## 5. Components

### 5.1 `findajob.admin.stack_discovery`

```python
@dataclass(frozen=True)
class StackPath:
    handle: str          # "alice", "brock", ...
    root: Path           # /opt/stacks/findajob-alice
    db_path: Path        # /opt/stacks/findajob-alice/state/data/pipeline.db
    jsonl_path: Path     # /opt/stacks/findajob-alice/state/logs/pipeline.jsonl

def discover_stacks(stacks_root: Path) -> list[StackPath]:
    """Glob `findajob-*/state/` directories under `stacks_root`. Returns
    sorted by handle. Skips siblings that don't match the pattern (dozzle,
    archivebox, ...). Skips stacks missing the `state/` subdir."""
```

Pure FS enumeration. No I/O against DB or JSONL. Easy to unit-test with a tmpdir.

### 5.2 `findajob.admin.stack_health`

```python
@dataclass(frozen=True)
class StackHealth:
    handle: str
    last_triage_complete: datetime | None    # most recent `pipeline_complete` event
    last_triage_failed: datetime | None      # most recent `pipeline_terminated` event
    last_aichat_failure: datetime | None
    last_discovery_failed: datetime | None
    last_prep_failed: datetime | None        # `prep_failed` or `prep_failed_reset`
    triage_success_24h: int                  # count of `pipeline_complete` in last 24h
    triage_failure_24h: int                  # count of `pipeline_terminated` in last 24h
    stage_counts: dict[str, int]             # {"scored": 42, "manual_review": 8, ...}
    stuck_prep_count: int                    # prep_in_progress age > 60min
    db_missing: bool
    jsonl_missing: bool
    error: str | None                        # short message if gather failed unrecoverably
    freshness: Literal["fresh", "late", "stale", "unknown"]

def gather(stack: StackPath, *, now: datetime | None = None) -> StackHealth:
    """Reads pipeline.db (mode=ro) and tails pipeline.jsonl. Returns one
    dataclass with everything the template needs. Catches per-stack
    exceptions and surfaces them via `error` field rather than raising.
    `now` injectable for testing."""
```

All time math accepts an injectable `now` for deterministic tests.

### 5.3 `findajob.admin.jsonl_tail`

```python
def tail_events(path: Path, *, max_bytes: int = 1_048_576) -> Iterator[dict]:
    """Read last `max_bytes` of `path`, yield decoded JSON events
    newest-first. Skips malformed lines (logs WARNING, doesn't raise).
    Returns empty iterator if file missing or empty."""
```

Bounded tail prevents pathological reads on long-running stacks. 1MB buffer holds ~1k–5k events depending on density — comfortably more than 24h on any active stack.

### 5.4 `findajob.web.routes.admin_stacks`

One route, ~30 lines:

```python
router = APIRouter()

@router.get("/admin/stacks/", response_class=HTMLResponse)
def stacks_index(request: Request):
    stacks = discover_stacks(Path("/opt/stacks"))
    health = [gather(s) for s in stacks]
    return templates.TemplateResponse("admin/stacks_index.html", {
        "request": request,
        "health": _sort_operator_first(health),
        "rendered_at": datetime.now(UTC),
    })
```

No business logic; orchestration only.

### 5.5 Template + nav changes

`templates/admin/stacks_index.html` extends `base.html`. One `<table>` with seven columns (see §6 below). Empty-state banner above the table when `health == []`. Per-row error renders the `error` string in the row.

`_nav.html` gains two conditional bits, both gated on a request-context `operator_mode` flag (passed from `app.py` once at app startup, read from env):

```jinja
<nav class="{% if operator_mode %}bg-rose-700{% else %}bg-slate-800{% endif %} ...">
  ...
  {% if operator_mode %}<a href="/admin/stacks/">Admin</a>{% endif %}
</nav>
```

Operator's red-bar visual cue applies on every page (board, materials, ingest, stats, admin), not just `/admin/stacks/`. Ambient mode indicator. Tester stacks render the existing color.

## 6. Template — column layout

| # | Column | Source | Render |
|---|---|---|---|
| 1 | Handle | `StackPath.handle` | `brock` (operator-first), then alphabetical |
| 2 | Last triage | `last_triage_complete` | Relative time + freshness badge: `3h ago` (green), `26h ago` (yellow), `2d ago` (red), `—` (unknown) |
| 3 | Stage distribution | `stage_counts` | Compact inline: `42 scored / 8 review / 0 prep / 5 applied / 1 interview / 0 offer / 12 rejected`. Zeros suppressed. |
| 4 | Stuck prep | `stuck_prep_count` | Number; bold red if > 0; `0` otherwise |
| 5 | 24h triage | `triage_success_24h / triage_failure_24h` | `12 ✓ / 0 ✗`. Failure count red if > 0. |
| 6 | Last failure | most-recent of `last_aichat_failure` / `last_discovery_failed` / `last_prep_failed_reset` | Relative time + event-type tag, or `—` |
| 7 | Drill-down | shell recipe | Pre-formatted `ssh docker.lan tail -F /opt/stacks/findajob-{handle}/state/logs/pipeline.jsonl`. Click-to-copy via Alpine.js (existing dep; convention used elsewhere in the app). |

Color tokens come from existing Tailwind tokens already in use elsewhere in `findajob.web` — no new design language.

## 7. Error-handling matrix

| Failure | Behavior | Observable as |
|---|---|---|
| `/opt/stacks` not mounted into container | Page renders empty-state banner | Banner: `No stacks found at /opt/stacks/findajob-*/. Check operator compose mounts.`; route returns 200 |
| Stack dir missing `pipeline.db` (mid-onboarding tester) | Row renders with `db_missing=True` | All cells `—`; info icon |
| DB query raises | Row renders `error` string | Per-row error visible; full page still loads |
| JSONL missing or empty | Event counts `0`, last-event timestamps `—` | Indistinguishable from "no events"; intentional |
| Malformed JSONL line | Skipped silently; one WARNING in `pipeline.jsonl` | Operator can grep their own logs |
| Operator mode flag unset | Route module not imported | 404 |
| Operator mode flag set, auth env unset | Route exists, public | Pre-existing app-wide behavior, not new |

## 8. Testing strategy

Matches existing `findajob.web` patterns — fixture-based, no real network or container.

| File | Coverage |
|---|---|
| `tests/test_admin_stack_discovery.py` | Tmpdir with `findajob-foo/state/data/pipeline.db` + `findajob-bar/state/`, asserts `discover_stacks()` returns the right handles. Skips non-`findajob-*` siblings (`dozzle/`, `archivebox/`, etc.). Sorts alphabetically. |
| `tests/test_admin_stack_health.py` | Per-stack: empty DB; populated DB with mixed stages; `prep_in_progress > 60min` rows must count, `< 60min` must not; missing DB → `db_missing=True`; corrupt DB → `error` set, no raise. Each becomes a fixture under `tests/fixtures/admin_stacks/`. |
| `tests/test_admin_jsonl_tail.py` | Boundary: empty file; file < `max_bytes`; file > `max_bytes` (must read tail only, not whole file); event types classified correctly; malformed JSONL line skipped not raised. |
| `tests/test_admin_stacks_route.py` | TestClient hits `/admin/stacks/`; asserts: route exists when `FINDAJOB_OPERATOR_MODE=1`, returns 404 when unset; renders all stacks; renders empty-state banner; renders per-row error without crashing the page; integrates with existing `BasicAuthMiddleware`. |
| `tests/test_admin_nav_color.py` | Asserts nav background class flips between operator-mode and tester-mode renders. May extend an existing nav test rather than spawn a new file. |

Fixture tree borrows the `tests/fixtures/onboarding/` pattern.

## 9. Observability

- One log line per request: `INFO admin_stacks: rendered N=6 stacks in 487ms`. Lets the operator grep `pipeline.jsonl` to confirm budget is being met. Useful when the tester wave grows past six.
- One log line per stack on `gather()` exception: `WARNING admin_stacks: gather failed for {handle}: {exc}`. Surfaces in operator's `pipeline.jsonl` and is also shown inline in the row.
- No new ntfy hooks. Push alerts on stale stacks belong in a follow-up `notify.py health-check` extension.

## 13. Decisions made during implementation

### 13.1 Route registration site (amends §4.1)

The plan moves the conditional `include_router` from `findajob.web.routes.__init__` to `findajob.web.app.create_app()`. Reason: registration inside `routes/__init__` evaluates the env at module-import time, which is global and one-shot. Tests that monkeypatch `FINDAJOB_OPERATOR_MODE` per-test cannot rebuild the route after the package is loaded the first time. `create_app()` runs once per app instance and reads the env at construction time — tests get clean isolation by setting/unsetting the env around `create_app` calls. The spec's intent is preserved (env-flag-gated, route module not imported when flag unset); only the hook moves. See plan §"Architecture" deviation note.
