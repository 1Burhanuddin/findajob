## 1. Problem statement

The post-#283 onboarding interview now collects an optional RapidAPI key in Step 1 and walks the candidate through a four-source briefing in Phase 1 + Section 3g. But the briefing closes with an unresolved promise — *"today the pipeline uses one specific RapidAPI service; a future version will help you pick the one that best fits your field."* The pipeline still uses a single hardcoded RapidAPI service (`jobs-api14`, LinkedIn-skewed) regardless of whether that's the right fit for the candidate's field. For testers in fields with light LinkedIn presence (social services, skilled trades, healthcare-niche), `jobs-api14`'s 0.12% hit rate (#274) means RapidAPI is functionally dormant on their stacks — they paid the API-key-collection friction at onboarding for zero recall lift.

The single-feed assumption is also baked into `findajob.fetchers` itself: `fetch_jobsapi_jobs()` is hardcoded to `jobs-api14`'s request shape, response shape, and `RAPIDAPI_KEY` env var. Adding a second RapidAPI feed today (e.g. JSearch — #310) means duplicating the function, hardcoding triage wiring for the second source, and double-touching `data/.env` for a separate key var. Two feeds becomes intractable; three never happens.

This issue closes both gaps in one body of work: the architecture (pluggable adapter framework) and the UX (picker step in the interview that recommends a feed for the candidate's field).

## 2. Goal

A candidate finishing onboarding has a job-source ingestion path tailored to their field, with credentials wired up and a successful connection test in their hand. Specifically:

- The interview reads a curated table of recommended feeds per candidate class and presents a profile-grounded recommendation with rationale, cost, and the option to override.
- The pipeline's RapidAPI ingestion is pluggable: each feed is a `JobSourceAdapter` subclass; `triage.py` iterates a registry instead of calling hardcoded fetcher functions; adding a new feed = one new adapter file + one registry line.
- A per-stack `config/active_sources.txt` declares which adapters are turned on. Each adapter declares its own env var (`JOBS_API14_KEY`, `JSEARCH_API_KEY`, etc.).
- The candidate's chosen feed has its credentials live-tested before onboarding completes — every query in their `jsearch_queries.txt` runs against the chosen feed, with streaming progress and honest framing ("the nightly fetch is the real test").
- Existing operator and tester stacks pulling v0.14 keep working without intervention via a one-time idempotent migration of `RAPIDAPI_KEY` → `JOBS_API14_KEY`.
- The framework generalizes beyond RapidAPI: future direct adapters (Workday CXS — #248, Gem GraphQL — #249) slot into the same `JobSourceAdapter` contract.

The bar is field-agnostic: a social-services candidate, a skilled-trades candidate, a healthcare-clinical candidate, and a corporate-tech candidate all finish onboarding with a feed appropriate to their field, not a hardcoded LinkedIn-skewed default.

## 3. Out of scope

Deferred to follow-on issues — not part of this design:

- **Migrating non-RapidAPI fetchers** (Greenhouse / Ashby / Lever / Gmail) **to the new framework.** Tracked in #410. Those fetchers stay function-style in this PR; only RapidAPI-flavored sources adopt `JobSourceAdapter` initially. Two patterns coexist transitionally — the migration happens after the framework has soaked in production for a few weeks.
- **Live Sonar-recommender for the picker** — tracked in #411. The picker uses a static `config/rapidapi_feeds.yaml` curation table; the live Perplexity-Sonar recommender that consumes the candidate's profile + adapter metadata is a follow-up that doesn't land until the static path has dogfood time.
- **Removing the operator's personal RapidAPI key from all tester stacks** (#412). This is operational, not code — coordinating with each tester to migrate to their own credentials. The rename `RAPIDAPI_KEY` → `JOBS_API14_KEY` is a *precondition* for a clean rotation, not the rotation itself.
- **Adzuna direct adapter.** Independent of this work; spin up when an Adzuna key is provisioned.
- **Workday CXS / Gem GraphQL adapters** (#248 / #249). Both will use `JobSourceAdapter` once landed; their adapter-implementation work is out of scope here.
- **Data-driven curation from production hit-rates** (#379). Replaces the static YAML over time once cross-user signal aggregation is in place.
- **Per-feed cost / quota tracking.** Future work, plausibly under the cost-observability epic (#240). Static curation surfaces free/paid-tier numbers but doesn't measure usage.
- **Per-feed re-run affordance** (change feed without re-onboarding). Slot in `/tools/` per #150; not in this PR.
- **OAuth flows for any feed** — out of scope.

## 4. Architecture

### 4.1 `JobSourceAdapter` framework

A new package `findajob.fetchers.adapters` introduces the contract every RapidAPI-flavored source implements. Each adapter is a standalone module that declares its registry name, display name, required env vars, and three methods (`is_configured`, `fetch`, `live_test`). The framework explicitly does not couple RapidAPI specifics into its base class — the same `JobSourceAdapter` Protocol is reusable by future direct adapters (#248 Workday, #249 Gem, future Indeed-direct, etc.).

**Package-layout note.** Today `src/findajob/fetchers.py` is a single module. To house the adapter subpackage, this PR converts it to a package: `src/findajob/fetchers/__init__.py` retains every currently-exported function (`fetch_jobsapi_jobs`, `fetch_greenhouse_jobs`, `fetch_ashby_jobs`, `fetch_lever_jobs`, `fetch_gmail_linkedin_jobs`, plus all helpers — they move into `__init__.py` or a sibling `_legacy.py` re-exported from `__init__`). Existing import paths (`from findajob.fetchers import fetch_greenhouse_jobs`, etc.) continue to work unchanged. The `fetchers/adapters/` subpackage is the new home for `JobSourceAdapter`-based code; nothing in the function-style code path moves until #410.

```
src/findajob/fetchers/
├── __init__.py                  # re-exports existing function-style fetchers (UNCHANGED API surface)
├── _legacy.py                   # (optional) — body of the old fetchers.py if __init__.py grows uncomfortable
└── adapters/
    ├── __init__.py              # re-exports the registry entry point
    ├── base.py                  # JobSourceAdapter Protocol + LiveTestResult dataclass
    ├── jobs_api14.py            # JobsApi14Adapter (refactor of current code)
    ├── jsearch.py               # JSearchAdapter (NEW; bundles #310)
    └── registry.py              # REGISTERED_ADAPTERS list + iter_configured_adapters()
```

`base.py`:

```python
from typing import Protocol, runtime_checkable

@runtime_checkable
class JobSourceAdapter(Protocol):
    name: str                            # registry identifier — matches config/active_sources.txt entries; e.g. "jobs-api14"
    display_name: str                    # user-facing — e.g. "Jobs API (jobs-api14)"
    source_label: str                    # value written to jobs.source DB column; preserves existing labels for backwards compat (e.g. "jobsapi_linkedin")
    required_env_vars: tuple[str, ...]   # e.g. ("JOBS_API14_KEY",)

    def is_configured(self) -> bool: ...
    def fetch(self, queries: list[str]) -> list[dict]: ...
    def live_test(self, queries: list[str]) -> "LiveTestResult": ...
```

`LiveTestResult` is a dataclass with `ok: bool`, `per_query: list[QueryResult]`, `auth_error: str | None`, and `bucket: Literal["auth", "rate_limit", "server", "network", "zero_rows", "mixed", "success"]` — the bucket maps directly to the user-facing message rendered by the feed-config form.

### 4.2 Registry + active-source resolution

`registry.py` exports an explicit list of registered adapter classes plus the runtime entry point used by `triage.py`:

```python
REGISTERED_ADAPTERS: list[type[JobSourceAdapter]] = [
    JobsApi14Adapter,
    JSearchAdapter,
]

def iter_configured_adapters() -> Iterator[JobSourceAdapter]:
    """Yield adapter instances that are active for this stack and configured."""
    active_names = _read_active_sources()
    for cls in REGISTERED_ADAPTERS:
        if cls.name not in active_names:
            continue
        instance = cls()
        if not instance.is_configured():
            log_event("adapter_not_configured", adapter=cls.name)
            continue
        yield instance
```

`_read_active_sources()` parses `config/active_sources.txt` (one adapter name per line; `#`-prefixed comments and blank lines ignored). If the file is missing, returns `["jobs-api14"]` — the backwards-compat default for existing stacks pulling v0.14 without re-onboarding.

`triage.py` becomes:

```python
for adapter in iter_configured_adapters():
    rows = adapter.fetch(queries)
    persist_rows(rows, source_label=adapter.source_label)
```

### 4.3 Per-adapter env vars

Each adapter declares its own env var(s) via the `required_env_vars` class attribute. No global `RAPIDAPI_KEY`. Concretely:

- `JobsApi14Adapter`: `name="jobs-api14"`, `source_label="jobsapi_linkedin"` (preserves existing DB rows' label so historical filters still work), `required_env_vars=("JOBS_API14_KEY",)`
- `JSearchAdapter`: `name="jsearch"`, `source_label="jsearch"` (new label; #310 AC #7 adds the value to the board filter dropdown), `required_env_vars=("JSEARCH_API_KEY",)`

Existing stacks have `RAPIDAPI_KEY` populated. A one-time idempotent migration in the entrypoint (`findajob.onboarding.injector.migrate_rapidapi_key_env`) copies `RAPIDAPI_KEY` → `JOBS_API14_KEY` in `data/.env` on first boot of v0.14, then removes the old line. Idempotent — safe to run on every boot.

### 4.4 Curation table — `config/rapidapi_feeds.yaml`

Operator-curated, human-readable, machine-parseable. Ships as `.yaml.example` (gitignored real file per the #84 mechanism-not-data convention). Five candidate classes × one recommended adapter each + a `default:` field for "no confident match." Schema:

```yaml
default: jobs-api14

classes:
  - name: corporate-tech
    description: "..."
    recommended_adapter: jobs-api14
    rationale: "..."
  - name: healthcare-clinical
    ...
  - name: skilled-trades-regional
    recommended_adapter: jsearch
    ...
  - name: social-services-nonprofit-education
    recommended_adapter: jsearch
    ...
  - name: remote-only-digital-nomad
    recommended_adapter: jsearch
    ...

adapters:
  - name: jobs-api14
    display_name: "Jobs API (jobs-api14)"
    rapidapi_url: "https://rapidapi.com/Pat92/api/jobs-api14"
    free_tier: "150 calls / month"
    paid_tier: "$5–25 / month for higher quotas"
    required_env_var: JOBS_API14_KEY
    coverage:
      best_for: "..."
      worst_for: "..."
  - name: jsearch
    ...
```

The interview prompt (Section 3h) is told explicitly: *"Only recommend adapters listed in `config/rapidapi_feeds.yaml`. Do not invent. If no class confidently matches the candidate, fall back to the `default:` field."*

### 4.5 Onboarding interview prompt — Section 3h

A new sub-phase inserted between the current 3g (source selection) and Phase 4 (translation/review). Fires only if the candidate's 3g selection includes `a` (paid service). Reads `config/rapidapi_feeds.yaml`, matches the candidate's profile (target roles, named target employers, geography, voice signals from Phase 2/3) to the closest class, presents the recommendation in plain language with adapter cost + rationale, and asks the candidate to confirm or override.

Emission: one new file block `<<<FILE: rapidapi_feed.txt>>>` containing a single line — the registry name of the chosen adapter (e.g. `jobs-api14` or `jsearch`). If the user declines to use any RapidAPI feed at this step, no block is emitted (the 3g selection collapses from `a, …` to `…`).

### 4.6 Parser + injector wiring

**`findajob.onboarding.parser`**: add `rapidapi_feed.txt` to `OPTIONAL_FILENAMES`; the parser is selection-agnostic per the #283 conditional-emission convention.

**`findajob.onboarding.injector`**:

1. New helper `migrate_rapidapi_key_env(env_path)` runs at app startup, before web routes register. Idempotent. See §4.3.
2. When parser captures a `rapidapi_feed.txt` block, injector writes its content to `config/active_sources.txt` (atomic, via temp-file + rename, same pattern as existing emissions).
3. After the interview emission completes and all files are written, the injector decides whether to gate on the feed-config form:
   - Read the chosen adapter's class via the registry.
   - If `adapter.is_configured()` returns True (env var already populated from Step 1), run the live test directly against the chosen adapter.
   - If the live test passes, write the sentinel and proceed to the board.
   - If the env var is missing OR the live test fails, redirect the user to `/onboarding/feed-config/{session_id}` instead of the sentinel.

### 4.7 Feed-config web route

**`findajob.web.routes.onboarding_feed_config`**: GET/POST `/onboarding/feed-config/{session_id}`.

The form is structured HTML, **not chat** — credential capture stays out of the LLM transcript. Form fields: a single password-input for the API key, with prose above explaining where to get it (specific to the chosen adapter, sourced from the YAML's `rapidapi_url` and `free_tier` fields).

POST handler:
1. Writes the submitted key to the chosen adapter's env var in `data/.env`.
2. Re-instantiates the adapter; calls `adapter.live_test(queries)` where `queries` is the contents of `config/jsearch_queries.txt` (parsed with the same blank-line + `#`-comment rules used elsewhere).
3. Renders the result via the same Jinja partial regardless of bucket — `_live_test_result.html` switches on `LiveTestResult.bucket`.
4. On `success` / `mixed` / `zero_rows`: shows the success/edge card with a "Finish onboarding" button. POST to `/onboarding/feed-config/{session_id}/finish` writes the sentinel.
5. On `auth` / `rate_limit` / `server` / `network`: shows the failure card with "Try again" (re-renders the form) and "Skip for now" buttons. Skip writes the sentinel; user can fill the key in later via `/config/`.

The route is idempotent against re-runs (`/onboarding/?mode=rerun` → re-route through 3h → re-route through `/onboarding/feed-config/` if needed).

### 4.8 Generality beyond RapidAPI

The `JobSourceAdapter` Protocol is RapidAPI-agnostic. Future adapters that don't go through RapidAPI (Workday CXS, Gem GraphQL, Indeed-direct, Adzuna-direct) implement the same Protocol with their own auth model. The picker UX in Section 3h is RapidAPI-flavored (RapidAPI's catalog is what makes signup-walkthroughs sensible), but the underlying registry and the `iter_configured_adapters()` entry point are source-agnostic. #410's eventual migration of Greenhouse/Ashby/Lever/Gmail into the framework operates on the same Protocol.

## 7. Error handling

| Surface | Failure | Behavior | User-facing message (plain language) |
|---|---|---|---|
| Section 3h | `rapidapi_feeds.yaml` missing/unparseable | Fall back to `default:` recommendation, still emit | "I don't have detailed coverage info right now, so I'll suggest the default option — Jobs API." |
| Section 3h | Class doesn't match any candidate class | Use `default:` field | "I don't have a confident recommendation for your specific field, so I'm going with **Jobs API** as the safe default." |
| Section 3h | User says "skip" | No `rapidapi_feed.txt` emitted; skip feed-config form | "No problem — we'll run without a paid service. You can add one later from `/config/`." |
| Injector | Adapter named in `rapidapi_feed.txt` not in registry | Refuse to write `active_sources.txt`; redirect to `/onboarding/feed-config/?error=unknown_adapter` (with manual dropdown) | "Something went wrong recording your feed choice. Pick from the list to continue." |
| Injector | `migrate_rapidapi_key_env` fails (file unwritable) | Abort entrypoint with structured log | Container exit, surfaces in `docker compose logs` |
| Live test | Auth fail on call 1 (401/403) | Stop loop; failure card | "Couldn't connect — JSearch didn't recognize your key. The most common cause is that your subscription isn't active yet." |
| Live test | Rate-limited mid-test (429 after call 1) | Stop loop; partial-result card | "Hit the free-tier rate limit during testing — that's normal, and your daily fetch will pace itself across the day." |
| Live test | Server error (5xx) | Stop loop; retry-or-skip card | "JSearch is having trouble responding right now. Try again in a few minutes, or skip and we'll try at the next daily fetch." |
| Live test | Network error / timeout | Stop loop; retry-or-skip card | "Couldn't reach RapidAPI. Check your internet connection or skip — we'll retry tomorrow." |
| Live test | All N queries return 0 rows, no errors | Edge card; user proceeds | "Connected — but none of your search terms matched today. That's normal for narrow searches. We'll keep checking daily; if it stays empty for a week, broaden your terms in `config/jsearch_queries.txt`." |
| Live test | Mixed (some success, some empty) | Success card with per-query counts | "Connected to JSearch. Across your N search terms, today returned X jobs total. Your full first batch lands tomorrow morning — that's the real test." |
| Triage runtime | `active_sources.txt` missing/empty | Default to `["jobs-api14"]` | (silent — backwards-compat) |
| Triage runtime | Active adapter env var blank | `is_configured()` returns False; skip with `log_event("adapter_not_configured")` | (silent — other adapters still run) |
| Triage runtime | Adapter `fetch()` raises | Log + continue (existing pattern) | (silent — other adapters still run) |
| Triage runtime | Cross-source duplicate rows | Existing `loose_fingerprint()` Tier-2 dedup | (silent — same as today) |

**Skip semantics:** "Skip for now" writes the chosen adapter to `active_sources.txt` and the sentinel, but leaves the env var blank. Triage's `iter_configured_adapters()` skips the unconfigured adapter and logs. The user can fill the key in later via `/config/` without re-running onboarding.

## 8. Testing

### 8.1 Unit tests (`tests/`)

- `test_adapter_registry.py` — `iter_configured_adapters()` filters by `active_sources.txt`, skips unconfigured, default fallback when file missing
- `test_jobs_api14_adapter.py` — mocked HTTP, request shape, response parsing, 429 retry, env var resolution (`JOBS_API14_KEY` only — see migration note below)
- `test_jsearch_adapter.py` — same surface, JSearch response shape, env var
- `test_live_test.py` — each failure bucket has a fixture: auth fail, rate-limit mid-test, server error, network error, zero-rows-everywhere, mixed success
- `test_migrate_rapidapi_key_env.py` — fixture `.env`s for: only `RAPIDAPI_KEY` set, only `JOBS_API14_KEY` set, both set, neither set, malformed file. Idempotent — running twice produces the same output
- `test_rapidapi_feeds_yaml.py` — schema validation, missing-class fallback, default field, malformed file fallback
- `test_active_sources_parser.py` — comments, blank lines, unknown adapters logged-and-skipped, default behavior

**Migration note for adapter tests:** `JobsApi14Adapter` reads `JOBS_API14_KEY` only — there is no production-code fallback to `RAPIDAPI_KEY`. The fallback exists only in `migrate_rapidapi_key_env()`, which runs at entrypoint *before* triage. After migration, `RAPIDAPI_KEY` is gone from `data/.env`. Test fixtures should reflect the post-migration state.

### 8.2 Integration tests

- `test_onboarding_flow_with_picker.py` — end-to-end through Section 3h: parser run against a captured emission corpus that includes `<<<FILE: rapidapi_feed.txt>>>`; injector writes `active_sources.txt`; sentinel-or-redirect logic verified against env state
- `test_feed_config_route.py` — GET/POST `/onboarding/feed-config/{session_id}`: form renders, submit triggers live test, success/failure paths render correctly, finish handler writes sentinel
- `test_triage_with_multiple_adapters.py` — `triage.py` with a fixture `active_sources.txt` containing two registered adapters; both run; rows persisted with correct `source` labels

### 8.3 Walkthrough (manual)

- **`findajob-test` walkthrough**: fresh-install onboarding, complete profile, pick `a` in 3g, observe 3h recommendation, complete picker, hit `/onboarding/feed-config/` (since `findajob-test` doesn't have a JSearch key by default), paste a real key, watch live test, finish. Confirm `config/active_sources.txt` and `data/.env` end up with the right values, sentinel writes, board reachable.
- **Existing operator stack** (any pre-v0.14 install with `RAPIDAPI_KEY` populated): verify entrypoint migration on first boot of v0.14. `RAPIDAPI_KEY` → `JOBS_API14_KEY` in `data/.env`; triage continues running unchanged; no re-onboarding needed.

### 8.4 Coverage exclusions

- No CI tests against the real RapidAPI free tier — too flaky, costs free-tier calls. All adapter tests use mocked HTTP.
- No unit tests for the LLM's adherence to the YAML — covered by the walkthrough.

**Cross-cutting non-functional requirements:**

- **Plain-language user-facing copy.** Picker chat, form, success/failure messages, edge cases. No technical jargon — say "your API key" not "X-RapidAPI-Key"; say "the free plan isn't subscribed yet" not "subscription not active."

- **Idempotent migrations.** Running the entrypoint multiple times is safe.

## 12. Migration

External users on v0.13 pulling v0.14:

1. `docker compose pull` + `docker compose up -d` — new image starts.
2. Entrypoint runs `migrate_rapidapi_key_env(data/.env)`:
   - If `JOBS_API14_KEY` already exists → no-op.
   - Else if `RAPIDAPI_KEY` exists → copy value to `JOBS_API14_KEY`, remove `RAPIDAPI_KEY` line.
   - Else → no-op (no key was set; user can fill in via `/config/`).
3. If `config/active_sources.txt` is missing → triage uses default `["jobs-api14"]`. No re-onboarding needed.
4. Optional: user can re-run `/onboarding/?mode=rerun` to pick a different feed via Section 3h.

The migration is invisible to the average user — no manual action is required. The CHANGELOG `### Migration required` bullet documents it for users who want to verify.
