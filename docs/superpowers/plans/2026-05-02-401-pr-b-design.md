## Task 3: Markdown rendering in assistant bubbles + `<<<FILE:>>>` block badging

**Files:**
- Modify: `src/findajob/web/templates/onboarding/_turn_bubble.html`
- Modify: `src/findajob/web/markdown.py` OR new `src/findajob/web/chat_render.py`
- Modify: `src/findajob/web/routes/onboarding_interview.py` (apply the helper before passing to template)
- Add: `tests/test_chat_render.py` (or extend existing markdown tests)

**Background:** `_turn_bubble.html` does `{{ content }}` (Jinja2 auto-escape). The user sees raw `**bold**`, `### headings`, `1.`/`2.` markdown lists, etc. Combined with the `<<<FILE: name>>>...<<<END FILE: name>>>` blocks emitted in Phase 5 (which can be tens of KB each), several turns of the walkthrough "vanished" off-screen as wall-of-text.

**Two-step solution:**
- Pre-process assistant content to replace each `<<<FILE: name>>>...<<<END FILE: name>>>` block with a small inline badge ("📄 Captured: {name}"). The parser still reads the raw blocks from the saved transcript on every turn — this is render-only.
- Then run the result through markdown rendering for the assistant role only.

**Important invariants:**
- The parser (`findajob.onboarding.parser`) reads from the *stored* assistant transcript in `session.history`, NOT from rendered HTML. This is why the FILE-block badging is render-only — the parser's input is unchanged.
- User bubbles render as plain text (auto-escape). Don't markdown-render user input — they didn't ask for it and it could surface unexpected formatting.
- The FILE-block badge name must be HTML-escaped (`html.escape`) — `name` comes from LLM output and could in principle contain `<`/`>`/`&`. The regex constrains `name` to non-whitespace non-`>` characters, but defense in depth.
- Per-task work splits cleanly: helper function in `markdown.py` (or new module), template wires it in.

**Steps:**
- [ ] Decide module boundary. `markdown.py` is the existing shared MD→HTML helper for materials + docs viewers. Adding a chat-specific function there is fine if it's clearly named (e.g. `render_chat_assistant_html`). Alternatively a thin new `chat_render.py` keeps domain-specific helpers separate. Author preference: extend `markdown.py` with a new exported function, keep imports tidy
- [ ] Implement `render_chat_assistant_html(text: str) -> str` in `markdown.py`:
  1. Run a substitution against the FILE-block regex. **Critical: compile with `re.DOTALL`** — block bodies always contain newlines (parser's regex has `(?P<body>.*?)\r?\n<<<END FILE:` which works because `.` doesn't span newlines but the explicit `\n` does; the render side wants `re.DOTALL` plus a non-greedy quantifier so the regex spans the whole multi-line block). Pattern: `re.compile(r"<<<FILE:\s*(?P<name>[^>\s]+)\s*>>>.*?<<<END FILE:\s*(?P=name)\s*>>>", re.DOTALL)`. Best practice is to import the parser's pattern (extract `findajob.onboarding.parser._BLOCK_RE` to a module-level constant if it isn't already exposed) so the two regexes can never drift; if extraction is awkward, document the duplication with a comment pointing at the parser source. Replacement: a placeholder of the form `<span class="captured-file inline-flex items-center gap-1 px-2 py-0.5 rounded bg-amber-50 border border-amber-200 text-amber-900 text-xs font-mono" title="Captured for the parser">📄 Captured: {html.escape(name)}</span>` — emit raw HTML; markdown's `md_in_html` extension preserves it
  2. Pass the result through `markdown.markdown(...)` with the same extension list as `render_markdown` (`fenced_code`, `tables`, `md_in_html`)
  3. Apply the same script-tag neutralization regex (`_SCRIPT_RE`) to the output. Skip the docs-rewriting / external-link rewriting paths — those don't apply to chat
- [ ] Modify `_turn_bubble.html`:
  - Continue auto-escaping for `turn.role == 'user'` (no markdown rendering on user input)
  - For `turn.role == 'assistant'`, render the pre-computed HTML via `{{ rendered_content | safe }}` — render the markdown server-side, NOT in the template
- [ ] Modify `routes/onboarding_interview.py`:
  - In `_render_chat` (around line 172): before passing `history` to the template, build a `rendered_history` list where each assistant turn has `rendered_content` set to `render_chat_assistant_html(turn.content)` and user turns pass through unchanged. Pass the rendered list to the template under the same key `history`
  - In `post_turn` (around line 278): apply the same to `assistant_message` so the per-turn HTMX response renders correctly. Pass `assistant_message_html` separately and let `_turn.html` use it for the assistant bubble
- [ ] Modify `_turn.html` (lines 13–14): use `rendered_content` for the assistant bubble. Update the partial's contract section in its docstring header
- [ ] Tests:
  - Add `tests/test_chat_render.py` covering: (a) basic markdown → HTML for headings/bold/lists, (b) FILE-block replacement with badge HTML, (c) script-tag neutralization, (d) html-escape of FILE block names containing special characters
  - Update `tests/test_web_onboarding_interview_routes.py` (or render-test file): assert the rendered HTML contains the `captured-file` class on a turn that emitted a FILE block, and that the raw `<<<FILE:` text does NOT appear in the rendered output
- [ ] **Verification:**
  ```bash
  uv run pytest tests/test_chat_render.py tests/test_web_onboarding_interview_routes.py tests/test_web_onboarding_interview_render.py -v
  # Manual: dev VM, complete a few interview turns including one Phase 5 emission. Assistant bubbles should show rendered markdown; FILE blocks should show as badges
  ```

## Task 15: Playwright walkthrough harness — autonomous replay against `findajob-test`

**Files:**
- Add: `scripts/walkthrough_harness.py`
- Add: `scripts/walkthrough_replay_corpus.py` (helper to extract user messages from a prior transcript)
- Add: `tests/test_walkthrough_harness.py` (unit tests for the parser/replay logic — NOT for the live browser run)

**Background:** The Task 16 walkthrough is the merge gate, but having the operator drive a 100-turn interview by hand every PR is unsustainable and introduces the operator as a source of variance ("did Brock answer differently this time?"). The harness drives the browser via Playwright, replays prior answers from `tmp/onboarding-walkthrough-2026-05-02/transcript.md`, captures DOM snapshots per turn, and emits a machine-checked findings report against the #401 acceptance criteria. This makes the merge gate cheap (~$3-6 of OpenRouter credit, ~1h walltime) and reproducible.

**Design decisions:**

- **Headless Playwright by default.** No display server needed on docker.lan / dev VM. If a selector misses, fall back to DOM snapshots and console-message inspection rather than headed mode (which requires the operator's remote-desktop session — keep it as escape hatch only).
- **Replay, not regenerate.** The harness loads `transcript.md` and extracts the `## Turn N — USER` blocks as the answer corpus. For each LLM question in the new walkthrough, it tries to match against the prior corpus by turn position; if the new conversation diverges (LLM asks an out-of-order question, or a new question Phase 4's proactive-categories revision introduced), the harness adapts using a small set of heuristics — never silently fabricates an answer
- **Adaptation rules (in priority order):**
  1. If the LLM's question text matches a prior question (>0.7 cosine similarity on the embed, or a strict substring-of-keywords match if no embed model is configured): use the prior answer
  2. If the LLM is asking the user to choose from a lettered list (Phase 4 revisions): the harness has a small `replay_intent.yaml` mapping prior categorical choices to canonical labels (e.g. "exclude sales" → "a/b/c letters that mention sales"); pick the matching letter(s)
  3. If the LLM appears to be asking a new question never seen in the prior transcript: log it to `findings.md` as `REVIEW` and emit a sentinel reply ("Skip — using prior context") so the conversation can continue
  4. If the LLM is asking for a confirmation ("ready to emit?", "next?"): always reply affirmatively
- **Per-turn assertions:** the harness checks each acceptance criterion at the appropriate turn boundary (e.g. after the kickoff turn, assert `#messages` has rendered markdown for `**` and `###`; after the first emission turn, assert no raw `<<<FILE:` text appears in the DOM but a `.captured-file` badge does; at finalize-ready, assert `#finalize-block` is populated WITHOUT a page reload). Each assertion writes a row to `findings.md`
- **Cost capture:** harness reads `data-cumulative-cost-usd` from the `#progress-row` after every turn. Final value goes into `cost-summary.json`. If final cost exceeds $7 (envelope upper bound + 20% buffer), the harness flags as `REVIEW` (not FAIL — could be normal variance) but stops further turns
- **Secret hygiene:** keys + HTTP Basic Auth credentials are loaded from `~/.secrets` (env-style file: `KEY=value` per line, optionally quoted) — operator maintains this file out-of-band; harness reads it but never logs the values. Recognized variables (all required except where noted):
  - `FINDAJOB_TEST_USER` — HTTP Basic Auth username for `findajob-test.<operator-domain>` perimeter
  - `FINDAJOB_TEST_PASS` — HTTP Basic Auth password
  - `FINDAJOB_TEST_OR_KEY` — OpenRouter key for the in-app interview chat (Step 1 form fill)
  - `FINDAJOB_TEST_RAPIDAPI_KEY` — RapidAPI key (Step 1 form fill)
  - `FINDAJOB_TEST_GOOGLE_KEY` — Google API key for RAG embeddings (Step 1 form fill, **optional**)
  Path is `~/.secrets` by default, override via `--secrets-file <path>`. The harness parses the file itself rather than `source`-ing it, to avoid shell injection from arbitrary file contents. The harness uses `FINDAJOB_TEST_USER:FINDAJOB_TEST_PASS` for Playwright's `httpCredentials` browser-context option (sent automatically on every request, no manual prompt-handling needed). The Step 1 form submission masks key values in DOM snapshots — `_keys_form.html` uses `<input type="password">` for these fields, but the `value` attribute IS captured by snapshot tooling, so the harness's snapshot serializer must redact `name="openrouter_api_key"` / `rapidapi_key` / `google_api_key` input values before writing snapshots to disk.
